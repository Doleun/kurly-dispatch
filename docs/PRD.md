# PRD — 택배 기사 관리 시스템

## 한 줄 정의

구글 시트로 하던 **휴무 계획 + 구역→기사 매핑**을 웹에서 처리하고, 매일 아침 **"10-1=누구"** 배차표를 즉시 볼 수 있게 하는 관리자용 도구.

## 문제

| 잘 되는 것 (시트) | 안 되는 것 (시트) |
|------------------|------------------|
| 요일별 휴무 분산 계획 | 구역번호 기준 당일 투입 기사 파악 |
| 수요 대비 휴무자 조절 | 10-1, 10-2, 20-1… 누가 가는지 찾기 |
| | 휴무로 빠진 구역 → 랜덤 투입 필요 여부 |

## 해결

1. 휴무/출근 관리 (요일 고정 + 특정일 1일 휴무)
2. 기사별 고정 구역·랜덤 후보 (1차/2차는 기사 속성, 구역은 한 목록)
3. 당일 배차표 자동 초안 (고정 채움 → 미배정 표시 → 후보군 수동 배치)
4. 역량/최대 처리량 메모 (배정 판단 참고)

## 범위 제외

- 마켓컬리 사이트 연동/전송
- 물량(건수) 단위 배차
- 성과 분석/교육 계획/맵 뷰 (Phase 2)
- **정산 실행·급여 계산 UI** (Phase 2 — 배차 데이터 틀은 Phase 1부터 확보)

## 사용자 · 권한 (확정)

| 역할 | 범위 | 설명 |
|------|------|------|
| **통합 관리자** | 전체 센터 | 대구·울산 등 모든 센터 조회·관리 |
| **센터 담당자** | 담당 센터 1개 | 자신의 지역(센터) 데이터만 관리 |

- 기사는 **센터 소속 1개 고정**
- 센터 담당자는 다른 센터 데이터 접근 불가

## 기사 · 컬리 아이디 (확정)

### 기본

- 각 기사는 **실명(정산용)** 과 **컬리 아이디**를 등록
  - 예: 오경인 = `Th300th1001`
- 컬리 사이트에서 실제 배차 입력 시 **아이디(및 해당 계정명)** 기준으로 배차

### 아이디 빌림 (대여)

| 상황 | 컬리 사이트 배차 | 우리 시스템 표시 (정산) |
|------|-----------------|------------------------|
| 본인 아이디 사용 | 본인 (예: 오경인 / Th300…) | 동일 |
| 아이디 미배정 → 타인 ID 빌림 | **ID 소유자** (예: 이순신) | **실배송자** (예: 홍길동) |

- 아직 컬리 아이디가 없는 기사는 **다른 기사의 아이디를 빌려** 배송할 수 있음
- **실배송자 ≠ 컬리 배차명** 이 될 수 있음
- 배차표에는 **두 정보를 항상 분리 저장·표시**
  - **컬리 배차용**: 어떤 아이디/계정명으로 컬리에 입력할지
  - **실배송자**: 정산·내부 관리용 실명

### 센터별 ID 매칭 (확정 — 2025)

| 항목 | 내용 |
|------|------|
| **기본 ID 매칭** | `driver_id_borrow_rules` — ID 없는 기사의 **편의용 기본 페어** (배차 초안 제안) |
| **당일 변경** | **대구·울산 모두** 배차(Phase 1d)에서 **구역별 ID를 매일 변경** 가능 |
| **센터별 고정 정책 없음** | 대구=고정 1:1 / 울산=당일 지정 같은 **센터 단위 규칙은 사용하지 않음** |

- 대부분 본인 ID 사용, ~5/50명은 타인 ID — **규칙은 초안 편의용**, 확정 전 언제든 덮어쓰기

### 예비 아이디

- 컬리에 **계정은 만들어져 있으나** 고정 구역·고정 배정이 없는 **예비(비상) ID**
- 평소에는 배차에 안 들어가고, **비상 투입** 시에만 선택
- 예: `아무개` 계정 — 긴급 시 배차표에서 지정

### 1개 ID → 여러 실배송자 (일일 물량 분할)

- **구역 단위가 아니라, 컬리 ID 하루 전체 물량**을 나눠 씀
  - 예: 홍길동 ID **당일 총 90건** → A·B·C가 **30·30·30**으로 분담
- **어느 구역을 누가 가는지는 매일 바뀔 수 있음** — 구역 배정과 ID 분할은 **별도 관리**
- **물량 입력은 컬리 사이트에서 수동** (본 시스템 범위 제외)
- 우리 시스템은 두 레이어로 기록:

| 레이어 | 내용 | 예시 |
|--------|------|------|
| **구역→ID 배차표** | 오늘 20-1, 20-2… 각각 **어떤 컬리 ID**로 넣을지 | 20-1=홍길동 ID, 20-2=이순신 ID |
| **ID 일일 분할** | 해당 ID **하루 전체**를 **누가 몇 건**씩 | 홍길동 ID 90 → A30 B30 C30 |

- 정산 Phase 2: **ID 일일 분할(`kurly_daily_splits`)** 기준으로 실배송자별 건수 집계

### 정산 (Phase 2 — 지금은 틀만)

- **구역 배정**과 **ID 일일 분할** 모두 확정 시 저장
- 1:1 빌림: ID 분할 1행 (actual 1명, 전량 또는 해당 몫)
- 다인 분할: `kurly_daily_splits`에 인원·건수
- 정산은 **실배송자 + share_quantity** (구역과 무관하게 ID 일 단위)

## MVP 성공 기준

- 매일 아침 배차표 확인 시간 단축
- 미배정/충돌 구역을 10초 안에 필터링 가능
- 확정 배차 이력 조회 가능
- **컬리 배차용 / 실배송자** 이중 표시로 컬리 입력·정산 준비 동시 가능

## 확정된 운영 규칙

| 규칙 | 내용 |
|------|------|
| 구역 | CRUD 가능, 배차표는 **등록된 활성 구역 전체** 기준 |
| 출근 | 휴무 제외 = 출근 |
| 무휴 희망 | 휴무 미지정 (배지 표시) |
| 단일 휴무 | 특정 날짜 1일만 |
| 1차/2차 | 기사 **속성** (default_time_slot). 고정 구역은 **한 목록** + 랜덤 후보. 2차→1차 지원은 **배차(Phase 1d)** |
| 1인 다구역 | 정상 (예: 10-1 + 10-2) |
| 기사 센터 | **1개 고정** |
| 컬리 아이디 | 대부분 본인 ID, 일부 타인 ID — **배차 때마다 변경 가능** (센터 무관, 규칙은 편의용) |
| 예비 ID | 비상 투입용, 고정 배정 없음 |
| 1 ID 다인 | **ID 하루 전체 물량** 분할 (구역과 별도) |
| 배차 표시 | **구역→ID 표** + **ID 일일 분할 표** |
| 정산 | 실배송자 기준 (Phase 2) |
| 랜덤 | 정해진 후보군에서만 |
| 충돌 | 같은 구역 2명 → 휴무 조정 필요 |

## 구역번호 체계 (확정)

- **센터별로 독립** — 대구 `20-1` ≠ 울산 `20-1`
- **등록은 한 줄** — `20-1`, `20-1가`, `20-1나` 각각 구역 하나로 등록
- 세분화 쓸 때: `20-1가` / `20-1나` 따로 등록
- 안 쓸 때: `20-1`만 등록
- **센터마다 ON/OFF 토글 없음** — 등록 방식으로 알아서 구분
